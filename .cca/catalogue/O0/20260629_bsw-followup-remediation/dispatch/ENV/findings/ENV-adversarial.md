## Verified ground truth

Single-writer today. `OrcaRig` (`web/app/components/scene/SalishScene.tsx:885`) builds its env via `makeRealWfxEnv` in a memo keyed `[gl, sun]` (`SalishScene.tsx:894-904`) and is the sole `scene.environment` writer: it assigns at `SalishScene.tsx:911`, restores the previous value under an identity guard at `913`, and disposes the handle at `914`. It is mounted unconditionally at `SalishScene.tsx:1633`.

Second bake today. `SliceRig` (`SalishScene.tsx:1114`) builds a SECOND env via `makeRealWfxEnv` in a memo keyed `[gl, sun]` (`SalishScene.tsx:1152-1162`), disposes it on unmount (`SalishScene.tsx:1163`), and passes it to `createOrcaPool({ env })` at `SalishScene.tsx:1342`. The pool-build effect dep array includes `env` (`SalishScene.tsx:1375`). `SliceRig` is mounted only when a station is selected (`SalishScene.tsx:1645-1646`). It never assigns `scene.environment` (confirmed by grep, see Q1).

Sun and renderer stability. `useScenicSun` is `useMemo(() => makeSun(SCENE_TIME, 48.5, -123), [])` (`SalishScene.tsx:709-711`). The dependency array is empty, so each call site holds a stable `sun` reference for the lifetime of that component, and both call sites compute from identical arguments. `gl` is the canvas renderer from `useThree`, stable for the canvas lifetime. So the `[gl, sun]` env memo in `OrcaRig` rebuilds only if `gl` itself changes (renderer recreation), not because the sun moved.

What the bake produces (`web/lib/scene/wfx/realWfxEnv.ts:51-98`): a PMREM of the decor sky dome built with `sunDirection` (`realWfxEnv.ts:53-61`), `pmremEnvironment = target.texture` (`realWfxEnv.ts:83`), sun fields, and `underwater` with `absorption` from `PROPOSED_RGB_EXTINCTION` (`realWfxEnv.ts:69-73`), `inScatterColor = lerp(WATER_TUNED_DEEP, WATER_TUNED_SHALLOW, 0.5)` (`realWfxEnv.ts:77-80`), `waterLevelY = opts.waterLevelY ?? 0`, and `visibility = opts.visibility ?? 1.5` (`realWfxEnv.ts:90-91`). `dispose()` frees both the render target and the generator (`realWfxEnv.ts:93-96`). `dispose` is optional on the type (`web/lib/scene/orca/materials/wfxEnv.ts:42`).

What the pool actually consumes. The pool threads `env` into per-instance `createOrcaController({ env })` (`web/lib/scene/reenactment/OrcaPool.ts:82-88`). The controller builds the material with `env` (`web/lib/scene/orca/OrcaController.ts:75-79`) and the eyes with `env` (`OrcaController.ts:83`). The skin material reads `material.envMap = env.pmremEnvironment` (`web/lib/scene/orca/materials/orcaMaterial.ts:61`) and the three underwater uniforms `uAbsorption`, `uInScatter`, `uWaterLevelY` (`orcaMaterial.ts:64-66`). The cornea reads `corneaMat.envMap = opts.env.pmremEnvironment` (`web/lib/scene/orca/eyes/orcaEyes.ts:62`). The pool never consumes `sunDirection`, `sunColor`, `sunIntensity`, or `visibility`. So pool lighting depends only on `pmremEnvironment` plus `absorption`, `inScatterColor`, `waterLevelY`.

Two consumer asymmetries that matter for disposal:
- The skin material has a `setEnv` that re-points `envMap` and copies the uniforms (`orcaMaterial.ts:140-146`), surfaced through `OrcaController.setEnv` (`OrcaController.ts:164-166`).
- The eyes have NO update path. `corneaMat.envMap` is assigned once at construction (`orcaEyes.ts:62`) and is never re-read. `OrcaController.setEnv` updates the material only, never the eyes.
- The pool path never calls `OrcaController.setEnv`. `createOrcaPool` builds controllers and runs `controller.update`, and `update` advances pose only (`OrcaController.ts:118-157`); no `setEnv` call exists in the pool. So both the pool skin material and the pool eyes pin the texture reference captured at build time, with no live re-subscription to the handle.
- `createOrcaPool().dispose()` disposes controllers only and never the env (`OrcaPool.ts:70-76, 134-136`). It does not own the texture.

## Q1 — Double-write failure mode and the guard

The locked invariant is exactly one `scene.environment` writer, `OrcaRig`. The consolidation does not add a writer, it removes a BAKE. The risk is that the additive accessor or the slice edit silently introduces a second assignment.

Concrete ways a second writer could sneak in:
1. Convenience write in the slice. A future edit to `SliceRig` adds `scene.environment = handle.pmremEnvironment` "to be safe" so the pool is lit even before `OrcaRig` runs. This would double-write and would race the `OrcaRig` effect on mount order, and would also create a competing dispose owner.
2. Accessor that writes on publish. If the ORCA-owned accessor (`web/lib/scene/orca/materials/wfxEnv.ts` extend, `web/lib/scene/orca/index.ts` export, per `wave_shape.yml:61`) does the `scene.environment` assignment itself "so consumers do not have to", the write moves out of `OrcaRig` and the single-writer location is no longer auditable at one site.
3. Accessor exposing a setter, not a value. If the published shape is a hook or object that hands consumers a way to set the scene environment (for example returning the `scene` plus the texture, or a `setEnvironment` callback), any consumer can become a writer. The handle must publish the value object only.
4. A React-context provider that wraps the scene and assigns `scene.environment` in an effect of its own, parallel to `OrcaRig`.

The guard (contract for ENV-Q and ENV-B): the accessor publishes the `WfxEnvHandle` object only, by ref or read accessor, with a lifecycle of valid-after-build and cleared-on-dispose, and it never reads or writes `scene.environment`. The slice consumes the published handle and passes it to `createOrcaPool` exactly as it passes its own bake today (`SalishScene.tsx:1342`), and adds no `scene.environment` assignment. `OrcaRig` keeps its assignment at `SalishScene.tsx:911` and its guarded restore at `913` unchanged.

Grep and AST check ENV-ADV must run:
- Assignment grep over the homepage scene graph: `rg -n 'scene\.environment\s*=' web/app/components/scene web/lib/scene`. The only assignment sites that may appear are `SalishScene.tsx:911` (the write) and `SalishScene.tsx:913` (the identity-guarded restore), both inside the `OrcaRig` effect. Any third assignment, in particular inside the new accessor file or inside `SliceRig`, is a P0 fail. Note that the sandbox route scenes assign `scene.environment` (`web/app/(sandbox)/slice/SliceScene.tsx:165`, `web/app/(sandbox)/reenactment/ReenactmentSandboxScene.tsx:213`, `web/app/workbench/WorkbenchScene.tsx:163`, `web/app/(sandbox)/orca/OrcaSandboxScene.tsx:95`). Those are separate scene graphs on separate routes and are out of scope for the homepage invariant. The grep must be scoped to the homepage scene tree, not the whole repo, or those hits must be explicitly excluded.
- Accessor purity check: `rg -n 'scene\.environment' <new-accessor-file>` returns zero, and `rg -n 'scene\.environment' SalishScene.tsx` restricted to the `SliceRig` body (`1114-` to the end of the slice block) returns zero.
- AST sense check: confirm the single writer is the `OrcaRig` `useEffect` keyed `[scene, env]` (`SalishScene.tsx:909-916`) and that no new component renders an effect that assigns `scene.environment`.

## Q2 — Disposed-texture read

After consolidation the pool borrows `OrcaRig`'s texture instead of owning its own. A disposed-texture read shows as black or garbage image-based lighting on the pool skin and eyes, often with a `GL_INVALID_OPERATION` or "texture has been deleted" warning in the WebGL console, because the cornea pins the texture at build (`orcaEyes.ts:62`) and the skin pins it unless `setEnv` is called, and the pool path never calls `setEnv`.

(a) `OrcaRig` env memo recompute replaces and disposes the texture while the pool still references it. Trigger: the `[gl, sun]` memo at `SalishScene.tsx:894-904` recomputes. With `sun` stable (empty-dep memo at `709-711`) and `SCENE_TIME` fixed, the only live trigger is `gl` changing, that is a renderer recreation such as a WebGL context loss and restore, a canvas remount, or a dev fast-refresh that rebuilds the `Canvas`. On recompute, the cleanup at `SalishScene.tsx:912-915` calls `env.dispose()` (`914`), which frees the old `target.texture` (`realWfxEnv.ts:94`). The pool controllers built against the old texture still point `material.envMap` and `corneaMat.envMap` at the freed texture. Symptom: black or garbage IBL on the pool plus a deleted-texture WebGL warning. Guard: re-key the pool on handle identity. The pool-build effect must depend on the published handle identity (replace `env` in the dep array at `SalishScene.tsx:1375` with the published handle or a readiness token), so a new texture forces a pool rebuild that re-reads `pmremEnvironment` at controller construction. ENV-ADV must prove handle identity is stable across the pool's lifetime under the gl-stable assumption, and must exercise a forced renderer recreation to confirm the pool rebuilds rather than reads stale.

(b) `OrcaRig` unmounts while `SliceRig` is still mounted. `OrcaRig` is mounted unconditionally (`SalishScene.tsx:1633`) and sits above the conditional `SliceRig` (`1645-1646`) in the same `Canvas`, so under normal navigation `OrcaRig` cannot unmount while `SliceRig` is mounted. The residual triggers are full-scene unmount such as a route change away from the homepage, a `Canvas` remount, and dev fast-refresh. On a full unmount React tears both down, and cleanup order between sibling subtrees is not contractually guaranteed to run `SliceRig`'s pool teardown before `OrcaRig`'s `env.dispose()` at `SalishScene.tsx:914`. If `OrcaRig` disposes first, the pool holds a freed texture for the interval before its own teardown. This is transient because nothing paints after a full unmount, so it is lower severity than (a), but it can still emit a WebGL warning. Guard: ownership and lifetime stay with `OrcaRig`, the pool never disposes the borrowed handle, and a teardown that nulls the pool's `envMap` references before the shared texture can be freed removes even the transient window. ENV-ADV should capture the dev fast-refresh and route-away sequences with the console open and confirm no deleted-texture warning.

(c) The slice retains its own dispose call. Trigger: ENV-INT deletes the second bake at `SalishScene.tsx:1152-1162` but forgets to delete the dispose effect at `SalishScene.tsx:1163`, or rewires the dispose to free the borrowed handle. Symptom: a double free of a borrowed handle. `SliceRig` unmounts on deselect while `OrcaRig` stays mounted, so the slice would free the texture that `OrcaRig` still owns and still has assigned to `scene.environment` (`911`), turning the WHOLE scene IBL black, not just the pool, and emitting a deleted-texture warning on the next frame. Guard: borrowed handles are never disposed by the borrower. ENV-INT must delete both the bake (`1152-1162`) and the dispose effect (`1163`) and add no replacement dispose. The contract is single-owner disposal, `OrcaRig` owns and frees, the slice and the pool only read.

## Q3 — Lighting-mismatch failure mode

Are the two bakes identical today? Yes, on every field the pool reads. Both call sites pass the same `gl`, the same `waterLevelY = SEA_LEVEL_Y`, and `sun` values from the same `makeSun(SCENE_TIME, 48.5, -123)` (`SalishScene.tsx:709-711`), even though each holds a distinct memoized object reference. Neither call site passes `visibility`, so both default to `1.5` (`realWfxEnv.ts:91`). `absorption` comes from the same module constant `PROPOSED_RGB_EXTINCTION` (`realWfxEnv.ts:69-73`) and `inScatterColor` from the same `lerp(..., 0.5)` (`realWfxEnv.ts:77-80`). The sky dome is built from the same `sunDirection` (`realWfxEnv.ts:54`). No divergence flagged. Additionally, `visibility` is inert for the pool: no pool consumer reads it (the skin uses absorption, inScatter, waterLevelY at `orcaMaterial.ts:64-66`, the eyes use only `pmremEnvironment`), so even a hypothetical visibility divergence could not change pool lighting.

envMapIntensity, exposure, tone mapping. `envMapIntensity` is per-material and a constant, `0.9` on the skin (`orcaMaterial.ts:57`) and `1.2` on the cornea (`orcaEyes.ts:60`). These are unaffected by which texture instance supplies `envMap`. Exposure and tone mapping live on the single shared renderer, so they are identical before and after. Sharing the texture cannot change intensity.

PMREM mip differences from a different generator instance. The mip chain is a deterministic function of the input scene and the generator settings. Identical input sky scene plus identical generator settings produce an identical pre-filtered mip chain, so the independent bake today and the shared bake after are byte-equal in content. Sharing one texture is strictly equal to or better than two equal bakes, never worse.

Parity expectation. The lit pool pixels are produced by identical envMap texels, identical underwater uniforms, identical material parameters, and the same renderer. So image-based lighting parity is expected to be exact at the shader-input level. The residual reasons a framebuffer diff is not literally bit-zero are not lighting: the pool is pose-animated, so frames must be compared at an identical playhead, and antialiasing and PNG encoding introduce sub-LSB jitter. Net expectation: exact at the IBL contribution, near-zero at the pixel level.

## Q4 — Parity proof metric (the bar)

GPU before and after capture, deterministic conditions. Use the homepage `?station` capture hook (`SalishScene.tsx:1788-1819`) and the render-host filename convention seen in `.cca/catalogue/O0/20260628_render-host/proof/`, where names encode `?station=lat,lng,Name` plus optional `&bsw_demo=N` and `&view=topdown` and a capture timestamp. Fix every variable that is not the change:
- Same station: `?station=48.5583362,-123.1735774,Orcasound%20Lab` (the Orcasound Lab coordinate already used in the proof set).
- Same demo count: `&bsw_demo=3` so the multi-orca pool fills the frame and any IBL difference has maximum surface area to show. `bsw_demo` maps to the labeled override at `SalishScene.tsx:1814-1818`.
- Same POV: the default dive-in (omit `view`), and a second pair at `&view=topdown` (`SalishScene.tsx:1806`) for an overhead read of the same pool.
- Same playhead. The slice opens on the presence-positive moment `SLICE_DEFAULT_T = 61.5` (`SalishScene.tsx:1074`), the pose comes from the timeline driver, so the before and after frames must be captured at the same playhead. Capture at the deterministic opening frame at `t = 61.5`, or pause the timeline at a fixed playhead before capture, so the diff measures lighting and not animation phase.
- Same renderer, same canvas size, same camera. Run both captures on the same render host with the same viewport so AA and tone mapping match.

Before is the current build with the two bakes. After is the ENV-INT build with the slice consuming the published handle and the second bake dropped. The "before" frame must already be a multi-orca presence-positive frame so the pool is visible in both.

The comparison. Read-examine both frames with the Read tool (the visual-verification rule), then compute a pixel metric over the pool region only, not the whole frame, since the change touches only the pool's IBL. Crop to the pool bounding box, in the deterministic capture the spawned orcas sit at the selected station, so the bbox is stable. Pass threshold: SSIM at or above 0.99 over the pool bbox, and mean per-pixel absolute delta within encoder and AA noise, on the order of a couple of 8-bit levels, with no localized region of saturated black or blown white that would signal a missing or wrong envMap. Exact-zero is not required because antialiasing and PNG encoding are not bit-deterministic frame to frame, but the delta should be near-zero because the texture content and uniforms are identical, so a non-trivial diff localized to the pool is a real regression, not noise. A useful sanity control: capture a third frame with a deliberately wrong env (for example the skin envMap left unset) to confirm the metric actually moves when lighting changes, so a near-zero pass is meaningful and not a stuck comparator.

Frame-time A and B. Dropping the slice's `makeRealWfxEnv` removes one `PMREMGenerator.fromScene` plus one `makeSkyDome` build and dispose at slice mount (`realWfxEnv.ts:53-61`) and one persistent PMREM render target held in VRAM until slice unmount. Measure slice-mount cost, the time from station select to first painted frame with the pool present, and steady-state frame time with the pool visible. Expected direction: a small improvement at slice mount, since one PMREM bake and one dome build are removed, plus lower VRAM, with steady-state frame time unchanged because per-frame cost is N rig skinning passes either way. The bar is parity, not speedup. Report the measured numbers, and a small regression in mount cost would itself be a flag worth a return.
